# BE-002 — Privacy not enforced in `search_users`; private users surfaced to strangers

**Severity:** High (privacy/contract gap)
**Component:** `app/services/search_service.py`
**Status:** CLOSED — fixed in `fix/testing/bugs` (PR #76, commit `cc5afea`). `UserRepository.search_by_name` now filters `User.is_private.is_(False)`. Private users excluded from search results. Verified: 515/515 tests pass.

## Summary

When a user toggles `is_private = true`, they expect to be hidden from non-followers in user search. Currently `SearchService.search_users` performs a simple `ILIKE` over `users.display_name`/`users.username` with no `is_private` filter. A stranger searching by handle still sees the private user in the results, defeating the privacy toggle.

## Reproduction

1. Register user `B` with username `private_target`.
2. As `B`: `PATCH /users/me/privacy` with `{ "is_private": true }`.
3. As an unauthenticated client (or user `A` who does not follow `B`):

```bash
curl http://localhost:8000/api/search/users?keyword=private_target
```

## Expected

`B` is **excluded** from results, OR the response includes a `is_private` / `is_pending` field so the client can render a "request to follow" CTA instead of a normal profile card.

## Actual

```json
{
  "success": true,
  "data": {
    "results": [
      { "username": "private_target", "display_name": "...", ... }
    ]
  }
}
```

Full profile preview is returned. The privacy toggle has no effect on search.

## Root cause

`app/services/search_service.py` — verified by:

```bash
grep -nE "is_private" app/services/search_service.py   # → 0 results
```

The search query has no `WHERE is_private = false` clause and the response shape includes no privacy hint.

## Suggested fix

Two viable options. Pick whichever matches the product intent:

**Option A — hide private users from non-followers entirely** (strict):

```python
# search_service.py
query = (
    db.query(User)
    .filter(User.username.ilike(f"%{keyword}%"))
    .filter(User.is_private == False)  # noqa: E712
)
```

**Option B — include them but mark them**:

Return `is_private: true` per result and let the client gate the UI. Requires updating `SearchUserItem` schema to include the field.

Either way, this should be a single decision documented in the API contract so the Cross-Platform and Frontend teams render consistent UX.

## Note

This bug interacts with **BE-003** (no follow-request semantics). Resolving them together makes the most sense.
