# BE-001 — `FollowerListData`/`FollowingListData` strip `private` field, privacy marker never reaches client

**Severity:** High
**Component:** `app/schemas/responses.py`, `app/services/follower_service.py`
**Status:** CLOSED — fixed in `fix/testing/bugs` (PR #76, commit `cc5afea`). `private: bool = False` added to both `FollowerListData` and `FollowingListData` in `app/schemas/responses.py`. Verified: `test_private_profile_followers_list_returns_private_marker` now passes.

## Summary

When a user has `is_private = true`, the follower/following service intentionally returns a `private: true` marker so the client can render a "this profile is private" empty state. The Pydantic response model strips the field before the response is serialised, so the client receives only `{ count, followers: [] }` — indistinguishable from a public user with zero followers.

## Reproduction

1. Create user `B` and set `is_private = true`.
2. Authenticate as user `A` (any other user).
3. `GET /users/{B_username}/followers`

```bash
curl -H "Authorization: Bearer $TOKEN_A" \
  http://localhost:8000/api/users/{B_username}/followers
```

## Expected

```json
{
  "success": true,
  "data": {
    "count": 5,
    "followers": [],
    "private": true
  }
}
```

## Actual

```json
{
  "success": true,
  "data": {
    "count": 5,
    "followers": []
  }
}
```

`private` key absent. Same pattern for `GET /users/{u}/following`.

## Root cause

- `app/services/follower_service.py:155-163` and `:226-234` correctly produce a dict containing `"private": True`:

  ```python
  if target_user.is_private:
      return {
          "success": True,
          "data": {
              "count": target_user.follower_count,
              "followers": [],
              "private": True,
          },
      }
  ```

- `app/schemas/responses.py:444` declares `FollowerListData` with no `private` field:

  ```python
  class FollowerListData(BaseModel):
      count: int
      followers: List[FollowerItem]
  ```

  Same omission on `FollowingListData` at `:461`.

- FastAPI's `response_model=FollowerListResponse` filters out keys not declared on the schema, dropping `private` before serialisation.

## Suggested fix

Add the field to both schemas:

```python
from typing import Optional

class FollowerListData(BaseModel):
    count: int
    followers: List[FollowerItem]
    private: Optional[bool] = None

class FollowingListData(BaseModel):
    count: int
    following: List[FollowingItem]
    private: Optional[bool] = None
```

`Optional[bool] = None` keeps the public-profile response unchanged (FastAPI omits `None` by default unless `response_model_exclude_none=False`), while letting the marker through for private profiles.


## Frontend impact

The Cross-Platform `ProfileScreen` and the Frontend `Profile` page have no way to distinguish "private" from "public with zero followers". The implication is silent UX loss: users see "0 followers" instead of "this profile is private".
