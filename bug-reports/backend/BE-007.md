# BE-007 — Upload-size violations return 400; HTTP recommends 413/422

**Severity:** Low (contract/HTTP-correctness, not a runtime bug)
**Component:** `app/services/track_service.py:245-249`, `app/services/user_service.py:268-272`, `app/services/playlist_service.py:340-344`
**Status:** Open (contract decision needed)

## Summary

Every "file too large" code path raises `HTTPException(status_code=400, ...)`. RFC 9110 specifies `413 Payload Too Large` for this exact case, and Pydantic-style validation errors typically use `422 Unprocessable Entity`. Returning `400` is not wrong, but it lumps oversize errors in with generic client errors, making it harder for clients to recognise and message specifically (e.g. "track too big — max 100 MB").

Verified locations using 400:

| Endpoint | File:line |
|---|---|
| `POST /tracks/` (audio file) | `app/services/track_service.py:245-249` |
| `POST /tracks/` (cover image) | `app/services/track_service.py:259-263` |
| `PUT /users/me/avatar` | `app/services/user_service.py:268-272` |
| `PUT /users/me/cover` | `app/services/user_service.py:268-272` (shared `_validate_image`) |
| `POST /playlists/{id}/cover` | `app/services/playlist_service.py:340-344` |

## Suggested fix

Pick one and apply consistently across all upload paths:

```python
# Option A — RFC-compliant
raise HTTPException(
    status_code=status.HTTP_413_REQUEST_ENTITY_TOO_LARGE,
    detail="File size must not exceed 100 MB",
)

# Option B — Pydantic-style validation error
raise HTTPException(
    status_code=status.HTTP_422_UNPROCESSABLE_ENTITY,
    detail=[{"loc": ["body", "file"], "msg": "File too large (max 100 MB)", "type": "value_error.size"}],
)
```

The current QA tests in `tests/api/test_oversize_uploads.py` assert `400` and have a TODO note (`# TODO clarify upload-size contract`) tracking this exact decision. Once the team picks a code, QA will flip the assertions to match.

## Frontend impact

Both Frontend (`upload-validation.spec.ts`) and Cross-Platform (`upload_flow_test.dart`) currently hard-code 400 in their error-mapping. Whichever code is chosen, the contract should be fixed and propagated to all clients in one go to avoid the worst case where one client maps 400 and another maps 413.
