# BE-004 — OAuth routes (`/auth/google`, `/auth/facebook`) have zero HTTP test coverage

**Severity:** Medium (regression risk on auth-critical endpoints)
**Component:** `app/routers/auth.py`, `tests/`
**Status:** Open (coverage gap, not a runtime bug)

## Summary

`POST /auth/google` and `POST /auth/facebook` are public, unauthenticated endpoints that mint full session tokens. There is no HTTP-level test for either. The team's recently added `tests/unit/test_auth_router.py` covers only `/auth/bootstrap-admin`. Service-level unit tests (`tests/unit/test_auth_service.py`) mock the verifier and bypass the FastAPI routing/validation layer entirely.

## Why this matters

These endpoints:
- Issue access + refresh tokens.
- Auto-register users on first call (`is_new_user` flag).
- Validate ID tokens / access tokens via external SDKs (Google, Meta).

A regression that, for example, bypasses ID-token verification or returns `access_token` for an unverified social identity would mint valid sessions for attackers. Catching that requires a TestClient-level test, not just a service-method unit test.

## Verification of the gap

```bash
grep -lE "/auth/google|/auth/facebook" tests/api/*.py tests/unit/*.py 2>/dev/null
# → no matches
grep -nE "google|facebook" tests/unit/test_auth_router.py
# → only line 45: /auth/bootstrap-admin
```

## Suggested coverage to add

Three minimal HTTP tests per provider (pseudocode, drop into `tests/api/test_oauth_router.py`):

```python
def test_google_login_invalid_token_returns_401(client, monkeypatch):
    monkeypatch.setattr(AuthService, "google_login",
        lambda db, req: (_ for _ in ()).throw(HTTPException(401, "invalid")))
    r = client.post("/auth/google", json={"google_token": "bad"})
    assert r.status_code == 401

def test_google_login_missing_token_returns_422(client):
    r = client.post("/auth/google", json={})
    assert r.status_code == 422
    assert any(err["loc"] == ["body", "google_token"] for err in r.json()["detail"])

def test_google_login_new_user_returns_is_new_true(client, monkeypatch):
    monkeypatch.setattr(AuthService, "google_login", lambda db, req: {
        "success": True,
        "data": {
            "access_token": "a", "refresh_token": "r",
            "is_new_user": True,
            "user": {"user_id": "...", "email": "...", "username": "..."},
        },
    })
    r = client.post("/auth/google", json={"google_token": "ok"})
    assert r.status_code == 200
    assert r.json()["data"]["is_new_user"] is True
```

Mirror for `/auth/facebook` with `facebook_token` field.

## Schema field names (for client teams)

Confirmed from `app/schemas/auth_schema.py`:

- Google: `{ "google_token": "<id_token>" }` (NOT `id_token`)
- Facebook: `{ "facebook_token": "<access_token>" }` (NOT `access_token`)

These names should also be documented in OpenAPI; current docstrings refer to `id_token`/`access_token` informally.
