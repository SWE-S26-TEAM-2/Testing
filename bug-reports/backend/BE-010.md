# BE-010 — `PATCH /users/me/username` accepts own username and returns 200 instead of 4xx

**Severity:** Medium (misleading API contract; "change" to same value silently succeeds)
**Component:** `app/routers/user_profile.py` — PATCH /users/me/username, `app/services/user_service.py`
**Status:** Open — discovered via real-backend Playwright run (2026-04-29).

## Summary

`PATCH /users/me/username` with the user's current username returns HTTP 200 instead of 409 or 422.

```bash
# Authenticated as mostafayasser
curl -s -X PATCH .../api/users/me/username \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"username":"mostafayasser"}'
# → HTTP 200 {"success":true,...}
```

## Why this is a problem

A username-change endpoint should reject a no-op change (same value) with **409 Conflict** ("username already taken" — by yourself) or **422 Unprocessable Entity** ("new username must differ from current"). Returning 200 makes the caller believe a change was made when nothing happened.

More critically: if the duplicate-check query uses `!= current_user_id` to exclude self, it would also allow *other* users to take a username that's exactly the current user's username — a potential race condition.

## Expected behavior

`PATCH /users/me/username` with the same username should return:
- **409 Conflict** (`"Username already in use"`) — consistent with how duplicate-username conflicts are handled for different users, or
- **200 OK** with a message like `"Username unchanged"` — if idempotent behavior is intentional (must be documented).

## Note

The QA test (`settings-real.spec.ts:63`) was specifically designed to trigger this path and assert a 4xx response. The test fails because the backend returns 200. If idempotent behavior is intentional, the test expectation should be updated — but the API contract must be documented either way.
