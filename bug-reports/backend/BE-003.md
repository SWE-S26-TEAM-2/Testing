# BE-003 — No follow-request semantics for private users; follow row is created immediately regardless of `is_private`

**Severity:** Medium (feature gap, privacy contract incomplete)
**Component:** `app/services/follower_service.py`, `app/models/follow.py`
**Status:** Open

## Summary

A common privacy contract on social platforms: when user `B` is private, user `A` cannot directly follow `B`; instead a *follow request* is created and `B` must approve it. The current implementation has **no pending/request state** — `POST /users/{B}/follow` immediately creates a `Follow` row whether `B` is private or not. The privacy toggle therefore prevents nothing meaningful.

## Reproduction

1. User `B` sets `is_private = true`.
2. User `A` (not following `B`):

```bash
curl -X POST -H "Authorization: Bearer $TOKEN_A" \
  http://localhost:8000/api/users/{B_username}/follow
```

3. Verify the follow record exists immediately:

```bash
curl -H "Authorization: Bearer $TOKEN_A" \
  http://localhost:8000/api/users/me/following
# → B is in the list.
```

## Expected (one of)

- **(a)** `202 Accepted` with `{ is_pending: true }` and a `FollowRequest` row that `B` must approve. `B` does NOT yet appear in `A`'s following list.
- **(b)** Documented decision that private profiles do not gate following at all (i.e. `is_private` only hides discovery and lists). Then close BE-002 + BE-003 with that decision and align UI copy.

## Actual

`200 OK` and the follow row is created. No request flow exists.

## Root cause

- `app/services/follower_service.py:20-50` (`follow_user`) creates the row unconditionally; the only branches are "self-follow → 400", "duplicate → 400", and "blocked → 403".
- No `FollowRequest` model. Verified:

  ```bash
  grep -rnE "is_pending|follow_request" app/models app/services | wc -l   # → 0
  ```

## Suggested resolution

Two possible directions; the team should pick one and document it in the contract that Frontend + Cross-Platform consume:

**Direction 1 — implement requests** (matches Instagram/Twitter private accounts):

- Add `app/models/follow_request.py` with `(requester_id, target_id, status, created_at)`.
- `POST /users/{u}/follow` on a private target → creates `FollowRequest` (status=`pending`), returns `202` with `{ is_pending: true }`.
- New endpoints: `GET /users/me/follow-requests`, `POST /users/me/follow-requests/{id}/approve|reject`.

**Direction 2 — document that follow is not gated by privacy**:

- Update API docs to clarify that `is_private` only affects search visibility and follower/following list visibility, not follow creation.
- Update Frontend/Cross-Platform UX copy accordingly so users are not led to expect a request flow.

Either resolution unblocks BE-002 and gives the client teams a definitive contract.
