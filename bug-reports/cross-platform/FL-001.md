# FL-001 — `feature/cross-tracks` changed Track model but left 9 compile errors in downstream files; all integration tests fail to compile

**Severity:** High (build-breaking; entire integration test suite and team's new unit tests cannot compile)
**Component:** `lib/screens/home/your_likes_card.dart:106`, `lib/screens/library/context_menu_sheet.dart:217-242`, `lib/screens/library/uploads_screen.dart:34,55`, `lib/screens/library/widgets/track_tile.dart:22,25`, `integration_test/player/mini_player_integration_test.dart:23`
**Status:** Open (verified post-merge via `flutter analyze`)

## Summary

`feature/cross-tracks` (merged in PR #48) changed the `Track` model in two breaking ways:

1. **`Track.id` renamed to `Track.trackId`** — the old `id` getter was removed.
2. **`Track.artist` changed from `String` to `TrackArtist?`** — the artist field is now a nullable object with `.displayName`, `.username`, etc.

These are correct API-alignment changes (the backend returns a nested artist object, not a flat string). However, not all screens were updated before merge. Result: 9 compile errors blocking the entire integration test suite and the team's own new unit tests (`test/liked_tracks_test.dart`, `test/widget_test.dart`).

## Why this matters

The QA widget tests (`test/widgets/`) still pass because they only import `mini_player.dart` and `social_buttons.dart`, which have no type errors. But every test file that triggers compilation of `context_menu_sheet.dart`, `track_tile.dart`, `uploads_screen.dart`, or `your_likes_card.dart` fails immediately with a compiler crash.

This blocks:
- All 13 integration test suites in `integration_test/`
- `test/liked_tracks_test.dart`
- `test/widget_test.dart`

## QA note

The `Track` model itself (`lib/models/track.dart`) is correct. The `artistName` and `formattedArtist` getters are already provided on `Track` to simplify migration. The fixes are mechanical — search for `track.artist` and `track.id` across the codebase and update.
