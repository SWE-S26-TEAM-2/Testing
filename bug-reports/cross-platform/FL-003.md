# FL-003 — Integration tests depend on in-memory mock user store and `Key('library.logout')` button; both removed in team's latest merge

**Severity:** Medium (integration test suite would fail at runtime even after FL-001 is fixed)
**Component:** `integration_test/helpers/app_test_helpers.dart`, `lib/screens/library/library_screen.dart`
**Status:** Open

## Summary

Two assumptions baked into `app_test_helpers.dart` no longer hold after the team's 219-commit merge:

### Issue A — `resetMockUsers()` resets an in-memory list that auth flows no longer consult

`app_test_helpers.dart:resetMockUsers()` clears and repopulates `mockUsers` from `mock_data/mock_users.dart`. This worked when `LoginScreen` / `SignupScreen` checked `kUseMockAuth` and called `MockAuthService` which read from `mockUsers`. After the merge, all auth screens call `authProvider.notifier.login/register/...` which delegates to the real Dio-backed `AuthService`. The `mockUsers` list is no longer consulted. When integration tests run:

- `loginAsSeededUser()` calls `actionButton('Log in')` and enters `test@gmail.com` / `12345678`
- `authProvider` fires a real HTTP request to the backend
- If the backend is unreachable (CI without a server, local dev without `docker-compose up`), the test hangs or crashes
- Even if reachable, `test@gmail.com` / `12345678` probably doesn't exist in the real DB → 401

### Issue B — `simulateLogout()` taps `Key('library.logout')` which no longer exists

```dart
// app_test_helpers.dart:244
Future<void> simulateLogout(WidgetTester tester) async {
  await openBottomTab(tester, 'Library');
  await tapKey(tester, 'library.logout');  // ← Key removed from LibraryScreen
  ...
}
```

`LibraryScreen` was rewritten in this merge from `ConsumerWidget` (with a logout `IconButton(key: Key('library.logout'))`) to a plain `StatelessWidget` with `onNavigate`/`onBack` callbacks. The logout button is gone. Any test that calls `simulateLogout()` will fail.

## Affected test suites

All suites that call `loginAsSeededUser()` or `simulateLogout()`, including at minimum:
- `integration_test/authentication/authentication_test.dart`
- `integration_test/library/library_integration_test.dart`
- `integration_test/navigation/navigation_integration_test.dart`
- `integration_test/settings/settings_integration_test.dart`
- `integration_test/stability/app_stability_integration_test.dart`

## Why this matters

After FL-001 is fixed, these tests will compile but fail at runtime for structural reasons unrelated to the actual feature being tested. Fixing them requires deciding on a test-auth strategy.

## Suggested fix

**Option A — Override `authProvider` for tests (recommended)**:

```dart
// integration_test/helpers/app_test_helpers.dart
import 'package:flutter_riverpod/flutter_riverpod.dart';
import 'package:my_project/providers/auth_providers.dart';

// A mock AuthNotifier that reads from the in-memory mockUsers list
class MockAuthNotifier extends AuthNotifier { ... }

// Wrap the app with provider overrides in launchApp():
Future<void> launchApp(WidgetTester tester) async {
  await resetMockUsers();
  await tester.pumpWidget(
    ProviderScope(
      overrides: [
        authProvider.overrideWith((_) => MockAuthNotifier()),
      ],
      child: const SoundCloudApp(),
    ),
  );
  ...
}
```

**Option B — Add logout back to LibraryScreen** (at minimum as a hidden test-only path):

Not recommended — pollutes production UI for test instrumentation.

**Option C — Use `--dart-define=USE_MOCK_AUTH=false` flow with a seeded test database**:

CI-friendly but requires a running backend. Add `docker-compose` to the CI workflow.

## Additional note

`kUseMockAuth` is still defined in `lib/main.dart` but is no longer read by any screen after the merge. It can be removed from `main.dart` once a test-auth strategy is settled.
