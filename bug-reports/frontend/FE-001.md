# FE-001 — Playwright mock cluster: LoginModal never advances to password step (39 tests blocked)

**Severity:** High (blocks 39 of 96 chromium-mock tests)
**Component:** `src/components/LoginModal/LoginModal.tsx`, `src/services/mocks/auth.mock.ts`, or `e2e/helpers/auth.ts` (root cause unconfirmed)
**Status:** Open

## Summary

39 chromium-mock Playwright tests across `auth-flows`, `auth-negative`, `navigation-stability`, `search`, `settings`, `settings-state`, `upload-flow`, `upload-validation`, and `upload-negative` fail with the same downstream symptom: the LoginModal does not transition from the email step to the password step (`signin` or `register`). Every spec that needs to seed an authenticated session via the modal therefore times out at `waitForPasswordStep`.

This single root cause is responsible for **all 39 failures** in the post-merge run. Identifying and fixing it should turn the suite green.

## Failing test cluster (39)

`auth-flows` (6), `auth-negative` (3), `navigation-stability` (6), `search` (13), `settings` (1), `settings-state` (1), `upload-flow` (1), `upload-negative` (4), `upload-validation` (1).

Full list available in `/tmp/fe-pw-merge.log` after running `npm run test:e2e:mock`.

## Reproduction

```bash
cd Frontend/media
npm run test:e2e:mock -- --grep "email input advances to password step for valid email format"
```

## Expected

After typing `test@example.com` and clicking Continue:
- Modal advances to the `signin` step.
- "Welcome back!" heading is visible (per `src/components/LoginModal/SignInStep.tsx:19`).
- Password input is visible.

## Update — post-merge artifact analysis revealed multi-root cluster

Triage of `test-results/auth-flows-*/error-context.md` shows two confirmed roots that account for at least part of the cluster. The third (mock-provider runtime selection) remains the leading hypothesis for the rest:

### Root A — OAuth Google button now lives inside Google's iframe (test rot)

`LoginModal.tsx:192` swapped the previously-plain `<button>Continue with Google</button>` for `<GoogleLogin />` from `@react-oauth/google`. That component mounts inside a cross-origin iframe. Playwright selectors targeting `button:has-text("Continue with Google")` no longer match.

Failing artifact: `auth-flows.spec.ts:74:7 › OAuth provider buttons are visible and clickable`.

```
Locator: locator('button:has-text("Continue with Google")')
Expected: visible
Error: element(s) not found
```

Decision needed (team):
- Keep Google's iframe widget AND expose a stable test hook on the wrapping `<div>` (`data-testid="oauth-google-iframe"` plus a documented frameLocator path), OR
- Roll a custom button driven by `useGoogleLogin()` so Playwright can interact directly.

### Root B — Authenticated landing-page redirect to `/discover` not firing

`auth-flows.spec.ts:136 › authenticated user is redirected from landing to /discover` seeds `auth_token` in `localStorage` via storage state, navigates to `/`, and expects URL to change to `/discover`. URL stays on `/`.

```
Expected: "http://127.0.0.1:3100/discover"
Received: "http://127.0.0.1:3100/"
Timeout:  10000ms
```

Likely caused by Header/landing rewrite in this batch of commits (notably `b0f2bab fix(fe): improve Header avatar persistence`). Verification needed in `src/app/page.tsx` (or the new landing component) — does the redirect logic still read `localStorage.auth_token` on mount?

## Investigation pointers (for whoever picks this up)

The transition is gated by `LoginModal.tsx:48-67`:

```tsx
const handleSubmit = async () => {
  if (step === "input" || step === "main") {
    ...
    try {
      setIsLoading(true);
      const { isExisting } = await AuthService.checkEmail(emailOrProfileUrl);
      setStep(isExisting ? "signin" : "register");
    } catch {
      setError("Something went wrong. Please try again.");
    } finally {
      setIsLoading(false);
    }
  }
}
```

The mock provider returns the right shape:

```ts
// src/services/mocks/auth.mock.ts:73-85
checkEmail: async (emailOrProfileUrl) => {
  await new Promise(r => setTimeout(r, 500));
  const existingAccounts = ["test@example.com", "soundcloud.com/testuser"];
  return { isExisting: existingAccounts.includes(emailOrProfileUrl) };
}
```

The Playwright config sets `NEXT_PUBLIC_USE_MOCK_API=true` for the chromium-mock project (`playwright.config.ts:67-72`). The 7-second timeout in `helpers/auth.ts:103` is much longer than the mock's 500 ms artificial delay.

Hypotheses to investigate, in order of suspicion:

1. **Mock provider not actually selected at runtime.** Verify by adding a `console.log` in `auth.mock.ts:checkEmail` and re-running one failing test. If the log never fires, the FE is loading `auth.api.ts` instead and a network call to a non-existent backend on port 3100 silently rejects.
2. **Continue button selector drift.** `clickContinue(page)` in `e2e/helpers/auth.ts` relies on a button locator. Recent header/banner UI rewrites may have introduced a second button matching the same text, splitting the click.
3. **Modal closing on outside-click before transition.** A new modal portal/backdrop in the rewritten Header may steal focus or fire a close event between Continue click and `setStep`.
4. **Hydration race.** The page navigates to `/discover` on auth detection; if a stale token is in storage when tests open the modal, the modal may unmount before `setStep` runs.

A 5-minute trace to capture the issue:

```bash
npx playwright test e2e/tests/auth-flows.spec.ts:42 --project=chromium-mock --trace on --headed
```

then open the trace via `npx playwright show-trace test-results/.../trace.zip`. The Network and Console tabs will reveal whether `checkEmail` fires and what it returns.

## Why this matters beyond the test count

Many of the 39 failing tests are scenarios that exercise authenticated flows (upload, settings, navigation past `/discover`). Their failure means the entire authenticated-user surface of the chromium-mock suite is currently unverified. Fixing FE-001 is the highest-leverage single change in the QA backlog.
