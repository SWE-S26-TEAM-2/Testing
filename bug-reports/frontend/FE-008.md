# FE-008 — E2E real-backend tests use wrong API response shapes (envelope + field names)

**Severity:** High (every E2E spec that creates/reads a track, playlist, or user silently fails or skips)
**Component:** `e2e/real/` — upload, track, playlist, search, settings, profile specs + `fixtures/api-client.ts`
**Status:** FIXED — all response paths corrected (`tests/fe-qa`, commit pending).

## Summary

The backend wraps every response in `{ success: bool, data: { ... } }`. The E2E helper
`apiRequest` returns the full JSON body as `.data`. As a result, accessing `.data.foo`
reads the envelope object, not the payload. Six files had systematic mismatches:

| File | Wrong access | Correct access |
|---|---|---|
| `upload-real.spec.ts` | `res.data?.id` / `body?.id` | `res.data?.data?.track_id` |
| `track-real.spec.ts` | `res.data?.id` / `list.data[0].id` | `res.data?.data?.track_id` / `data?.tracks[0].track_id` |
| `playlist-real.spec.ts` | `created.data?.id` | `created.data?.data?.playlist_id` |
| `settings-real.spec.ts` | `me.data?.bio` / `me.data?.is_private` | `me.data?.data?.bio` / `me.data?.data?.is_private` |
| `profile-real.spec.ts` | `r.data` (assumed array) / `.username` on follower | `r.data?.data?.followers` / `.user_id` |
| `search-real.spec.ts` | `Array.isArray(r.data)` | `r.data?.data?.tracks` is the array |

## Additional issues fixed in the same pass

### 1. `GET /tracks` endpoint does not exist
`track-real.spec.ts` and `playlist-real.spec.ts` both called `apiRequest('/tracks')`
as a fallback to find an existing track. This endpoint returns 404 — the track list
is at `GET /users/{username}/tracks` (`UserTracksResponse`). Fixed to call `/users/me`
first to get the username, then `/users/{username}/tracks`.

### 2. `FollowerItem` has no `username` field
`profile-real.spec.ts` polled follower list and compared `.username === seedUser.username`.
`FollowerItem` exposes `user_id` + `display_name`, not `username`. Fixed to match by `user_id`
via a second `/users/me` call.

### 3. Response type annotations in `api-client.ts`
`apiCreatePlaylist`, `apiUploadTrackMultipart`, `apiSearchTracks` all had incorrect generic
type parameters. Updated to reflect the actual envelope shapes.

## API shapes (confirmed from Backend schemas)

```
POST /tracks/           → { success, message, data: TrackCreateData { track_id, title, ... } }
GET  /tracks/{id}       → { success, data: TrackData { track_id, play_count, ... } }
GET  /search/tracks     → { success, data: SearchTracksData { tracks: [{ track_id, ... }] } }
POST /playlists/        → { success, message, data: PlaylistData { playlist_id, ... } }
GET  /users/me          → { success, data: UserProfileData { user_id, username, bio, is_private, ... } }
GET  /users/{u}/tracks  → { success, data: UserTracksData { user_id, tracks: [TrackData] } }
GET  /users/{u}/followers → { success, data: FollowerListData { count, followers: [{ user_id, display_name }] } }
```
