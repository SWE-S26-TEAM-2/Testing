# FE-005 — LoginModal has no forgot-password entry point; backend endpoint exists, UI does not expose it

**Severity:** Low (feature gap; not a runtime bug)
**Component:** `src/components/LoginModal/SignInStep.tsx`, `src/components/LoginModal/LoginModal.tsx`
**Status:** Open

## Summary

Backend exposes `POST /auth/forgot-password` with rate limiting (3 requests / 15 min) and `POST /auth/reset-password` to complete the flow. Both routes are tested and shipping. The frontend LoginModal does not expose any UI affordance to trigger them. Users who forget their password have no path to recovery from the web app.

## Verification

```bash
grep -rnE "forgot.password|forgot_password|reset.password" Frontend/media/src/components/LoginModal
# → 0 matches
```

```bash
# BE endpoints exist (verified from Backend repo):
# app/routers/auth.py:184 POST /auth/forgot-password
# app/routers/auth.py:205 POST /auth/reset-password
```

## Expected

LoginModal `signin` step should have a "Forgot your password?" link beneath the password input. Clicking should:

1. Open a new step that asks for the email (or pre-fill from the current input).
2. Call `AuthService.forgotPassword(email)` → POST `/auth/forgot-password`.
3. Show a generic success message ("If an account exists for that email, we sent a reset link.") regardless of whether the email is registered (BE already handles enumeration prevention).
4. Optionally surface 429 with a friendly "try again in a few minutes" copy.

## Actual

No link, no flow.

## Suggested implementation

Add a step `forgot-password` to the existing `step` union in `LoginModal.tsx:21`:

```tsx
const [step, setStep] = useState<
  "main" | "input" | "register" | "signin" | "tell-us-more" |
  "verify-email" | "forgot-password" | "forgot-sent"
>("main");
```

Add a `<button onClick={() => setStep("forgot-password")}>` to `SignInStep.tsx`, plus a `ForgotPasswordStep` component that calls `AuthService.forgotPassword` and transitions to `forgot-sent` on success.

The reset-password completion flow (clicking the link in the email) is a separate page (`/reset-password?token=...`) and should be tracked as a follow-up if not implemented.

## QA impact

Without this UI:
- Scenario E2 in [Testing/E2E_TEST_SCENARIOS.md](../../E2E_TEST_SCENARIOS.md) is BE-only (the FE half is intentionally skipped).
- The `auth-flows.spec.ts` cannot test the forgot-password user journey end-to-end.
