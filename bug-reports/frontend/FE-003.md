# FE-003 — Jest `Footer.test.tsx`: 5 audio/queue tests fail because jsdom does not implement `HTMLMediaElement`

**Severity:** Medium (5 tests permanently red; can hide real audio regressions)
**Component:** `src/__tests__/Footer.test.tsx`, `src/components/Footer/Footer.tsx`, jest config
**Status:** Open

## Summary

jsdom (used by jest) ships only a stub of `HTMLMediaElement`. `play()` / `pause()` are no-ops that do not update the element's internal state, and the `play` Promise resolves with no actual playback. The 5 Footer tests that try to verify audio synchronisation against a Zustand player store therefore observe state that never converges, and they time out at the `waitFor` boundary.

## Failing tests (5)

- `Footer Component (Global Player) › Volume Control › shows volume slider on hover`
- `Footer Component (Global Player) › Queue Drawer › displays queue items in drawer` (added by team post-merge)
- `Footer Component (Global Player) › Queue Drawer › highlights current playing track in queue` (added by team post-merge)
- `Footer Component (Global Player) › Audio Element Sync › plays audio when isPlaying is true`
- `Footer Component (Global Player) › Audio Element Sync › pauses audio when isPlaying is false`

The Volume + Queue tests share a different but related cause (selector / hover trigger missing in jsdom). The Audio Element Sync tests are the canonical jsdom-media issue.

## Reproduction

```bash
cd Frontend/media
npm test -- --testPathPattern=Footer.test.tsx
```

## Expected (Audio Element Sync test)

```
expect(usePlayerStore.getState().isPlaying).toBe(true)
```

## Actual

```
Expected: true
Received: false
- Timeout - waitFor(() => usePlayerStore.getState().isPlaying).toBe(true)
- HTMLMediaElement.prototype.pause has been called
```

## Root cause

jsdom (default jest env in this project) does not implement `HTMLMediaElement.play / pause / load / canPlayType` beyond stub methods that do nothing. The current Footer test does:

```tsx
HTMLMediaElement.prototype.play = jest.fn().mockResolvedValue(undefined);
HTMLMediaElement.prototype.pause = jest.fn();
```

The mock satisfies the call-count assertion (`toHaveBeenCalled`) but the player's internal state machine, which subscribes to `play`/`pause` events on the audio element, never advances because no event fires from the mocked methods.

## Suggested fix (pick one)

**Option A — accept jsdom's limits and assert on the mock, not the element**

Drop assertions that wait for the audio element's effect on the store. Verify only that the store's pause/play methods are wired and that the element's `play`/`pause` mocks are called:

```tsx
test("Footer triggers audio.play when store.isPlaying flips to true", () => {
  render(<Footer />);
  act(() => usePlayerStore.setState({ isPlaying: true }));
  expect(HTMLMediaElement.prototype.play).toHaveBeenCalled();
});
```

This trades behaviour-coverage for determinism. The audio→store roundtrip moves to Playwright (where Chromium implements the full media element).

**Option B — swap to Playwright + audio fixture**

These 3 specs become a single Playwright test using the real silent `sample.mp3` fixture in `e2e/real/fixtures/`. Real Chromium plays the file and the player store actually transitions.

**Option C — install `jest-environment-jsdom-sixteen` + `node-mocks-http` and patch the prototype**

Higher complexity, fragile across jest upgrades. Not recommended unless option A leaves a critical gap.

The QA position is **Option A is right**: jest is for unit/component coverage, Playwright is for browser-realistic media. The current tests promise a guarantee jest cannot deliver, so they are noise.

## Why this matters

These 5 tests are *permanently* failing in CI on every push. Permanent red noise trains reviewers to ignore CI status, masking real regressions when they appear. Fixing this (by either rewriting or moving to Playwright) restores the signal value of CI.
