# FE-009: Next.js Middleware Auth Guard blocks Playwright E2E Tests on Protected Routes

## 🐛 Bug description
Playwright E2E tests targeting protected routes (`/stream`, `/library`, `/settings`, `/notifications`, `/creator/upload`) are unexpectedly redirected to `/login` during CI/automated test runs. 

The tests themselves correctly inject the `auth_token` into `localStorage` and the `sc_auth_token` into the Playwright browser context cookies. However, the Next.js middleware fails to recognize the injected cookie in the headless browser environment, triggering a 307 redirect to the login page and causing 10 E2E tests to fail.

## 🔄 Steps to reproduce
1. Ensure Playwright is configured to target `https://streamline-swp.duckdns.org`.
2. Run the Playwright test suite (`npx playwright test`).
3. Observe failures in `feed.spec.ts`, `settings.spec.ts`, `library.spec.ts`, etc., specifically on the auth-protected landing tests.
4. Review the Playwright trace viewer: the browser successfully lands on `/stream`, but is immediately redirected by the Next.js `middleware.ts` to `/login`.

## 💻 Environment
- **Environment:** Playwright headless Chromium
- **File:** `media/middleware.ts` and `media/e2e/helpers/auth.ts`
- **Framework:** Next.js App Router

## 🎯 Expected behavior
The Next.js middleware should successfully read the `sc_auth_token` cookie injected by Playwright's `browserContext.addCookies()` and allow the headless browser to access protected routes, allowing the E2E assertions to run.

## 🛠 Possible Solution
1. **Middleware Environment Check:** Update `middleware.ts` to explicitly allow requests passing a specific E2E-only header (e.g., `x-e2e-bypass: true`) when `process.env.NEXT_PUBLIC_USE_MOCK_API === 'true'`.
2. **Playwright Config:** Add the bypass header to the global Playwright configuration `use: { extraHTTPHeaders: { 'x-e2e-bypass': 'true' } }`.
3. **Alternative Cookie Strategy:** Investigate if Playwright's cookie injection onto `streamline-swp.duckdns.org` is being rejected by the browser or Next.js due to cross-site origin issues or strict secure flags.
